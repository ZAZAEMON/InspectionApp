1. TOP-RIGHT TOOLBAR OVERLAP  (Shift / Auditor under the window buttons)
--------------------------------------------------------------------------------
Symptom : In the PPAP toolbar the "Auditor" text box overlapped the custom
          window caption buttons (minimize / maximize / close) at the very
          top-right, looking squashed and distorted.
Cause   : MainWindow floats its own caption buttons (3 x 46px = 138px wide) on
          top of the top-right corner of every hosted view. The PPAP and Home
          toolbars docked their Shift + Auditor group to the right edge with
          only an 8px margin, so the box slid underneath those buttons.
Fix     : Reserved the caption-button width on the right of those toolbars -
          right margin changed from "0,0,8,0" to "0,0,146,0" (138px + 8px gap).
Files   : Views/PPAPView.xaml
          Views/HomeView.xaml   (same latent bug, fixed at the same time)

--------------------------------------------------------------------------------
2. TOOLBAR BUTTONS HARD TO CLICK  +  TOP TEXT CUT OFF (when maximized)
--------------------------------------------------------------------------------
Symptom A: Toolbar buttons (Menu, Select Part Type, Report, Add, ...) only
           responded to clicks on a thin strip near their bottom edge.
Cause  A : The WindowChrome caption height is 32px; that top strip is a drag
           region that swallows mouse clicks. The toolbars sit directly under
           it, so only the part of each button below 32px was clickable.
Fix    A : Marked the interactive toolbar controls as hit-test-visible in the
           window chrome:
             - WindowChrome.IsHitTestVisibleInChrome="True" added to the shared
               ToolbarButtonStyle (fixes every toolbar button app-wide), and
             - the same on the Shift dropdown and Auditor box in PPAP and Home.
           Empty toolbar areas stay draggable, so the window can still be moved.

Symptom B: The top of the toolbar text was shaved off when the window was
           maximized.
Cause  B : A maximized WindowChrome window overflows the monitor edges by the
           resize-border thickness, pushing the top ~8px of content off-screen.
Fix    B : Added a WM_GETMINMAXINFO hook that constrains the maximized window to
           the monitor work area (fills the screen exactly, respects the
           taskbar, no clipping, no stray border).
Files   : App.xaml              (ToolbarButtonStyle)
          Views/PPAPView.xaml   (Shift / Auditor controls)
          Views/HomeView.xaml   (Shift / Auditor controls)
          MainWindow.xaml.cs    (WM_GETMINMAXINFO hook + interop structs)

--------------------------------------------------------------------------------
3. "PART LOADER" SEARCH BOX TEXT CLIPPED  (Product Audit view)
--------------------------------------------------------------------------------
Symptom : Text in the part-search box on the Product Audit screen was cut off
          at the top and bottom.
Cause   : The box was Height=30, but StandardTextBoxStyle adds 5px top/bottom
          padding plus a border (which grows to 2px on focus), leaving only
          ~16px of room for a 12px font line.
Fix     : Enlarged the box - Height 30 -> 40 and FontSize 12 -> 14 - giving the
          text comfortable vertical room.
Files   : Views/HomeView.xaml

--------------------------------------------------------------------------------
4. PPAP "ALL PARTS" REPORT - PARAMETERS MISALIGNED ACROSS PARTS
--------------------------------------------------------------------------------
Symptom : When exporting a PPAP report covering all parts, every part showed the
          SAME parameter list and the readings did not line up with the correct
          parameters. Each part actually has a different set (and a different
          number) of parameters.
Cause   : The export merged every session into ONE shared "Parameter" column,
          grouping all readings by serial number only and labelling each row
          with the first part's parameter name. Other parts' readings were then
          forced onto those rows by position.
Fix     : Reworked the Excel export so a multi-part report renders ONE stacked
          block per part, each self-contained:
             - a part banner  ("Part: <no>   (N session(s))")
             - its own header row with that part's sessions as columns
             - its own parameters as the rows
          Blocks are stacked vertically with a spacer between them. Single-part
          reports are unchanged.
Files   : Services/ExcelExportService.cs
